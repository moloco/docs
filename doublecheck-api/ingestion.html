Doublecheck Ingestion API

This document describes Moloco Doublecheck’s API for ingesting data.

Since Doublecheck joins information from multiple events, it is recommended to send events in chronological order by the time of occurrence of an event.

Definitions ¶

A product is a mobile app that an advertiser (or agency) is trying to promote. A product is identified by a product-id. It should be noted that a product-id per se may not uniquely identify a product. A product can be uniquely identified only if a product-id and account-id (defined below) are considered together.

An account represents an entity that tries to advertise a product. It corresponds to either an advertiser or an agency if it acts on behalf of an advertiser. An account is identified by an account-id.

A device represents a mobile device where a user can see an advertisement, click through it, and install the advertised product. The device is identified by a globally unique identifier, called a device-id or IDFA. Since it is useful to consider a device-id together with OS, this document also uses the term android-idfa or ios-idfa for the same entity.

A network is an entity that runs one or multiple campaigns for one or multiple accounts (advertiser/agency). A network is uniquely identified by a network-id. A typical example is a DSP. It is common that a network is paid when a conversion is attributed to its ad-event. For this reason, a network is motivated to maximize the number of conversions.

An advertisement represents a network’s attempt to achieve a conversion for a given product and is identified by an ad-id. In RTB (real-time bidding), an advertisement begins when a network places a bid, which creates a set of impressions and clicks. The bid and resulting impressions/clicks all belong to the same advertisement. Networks typically place multiple bids for the same product in an effort to achieve conversions, therefore, networks have many ad-ids for a given product. A tuple of ad-id, network-id, and product-id should be able to uniquely identify an advertisement.

An ad-site is where an inventory is located. A common term for describing an ad-site is a sub-publisher. Currently, an ad-site is a mobile-app that sells space for advertisements.

A sub-network is a channel that connects a network to an ad-site. In RTB, it is typically an ad exchange.

An event is a user’s action or its derivative; e.g.,

  • seeing an advertisement

  • clicking on an advertisement

  • installing a mobile app, which is captured by a tracker and sent to Doublecheck as a conversion event

Events are categorized into two kinds:

  • ad-event: ‘impression’, ‘click’

  • in-app-event: ‘install’

Each kind has its own endpoint for ingestion, as the required parameters are different. An event-id may be used to identify an event, but it can be left blank. In fact, in most cases, the sender of an ad-event is not capable of generating a unique event-id.

Identifiers ¶

Most identifiers (or values) sent in requests to Doublecheck are strictly checked against certain patterns. We define a few commonly used patterns, and their regular expression definitions are:

  • {safe-id}: ^[a-z0-9][_a-z0-9]{3,63}$

  • {unrestricted-id} ^[_a-z0-9]{0,64}$.

  • {app-id}: ^[a-z0-9][_a-z0-9-.]{3,63}$

Post an Event ¶

An endpoint in this section posts an event in Doublecheck’s queue. Doublecheck asynchronously handles requests to reduce client-side latency; i.e., it responds without waiting for its background worker to finish handling the posted event.

The POST method should be used because the operations are neither safe nor idempotent. All of the information for a posted event should be sent in the query parameter, for easier recovery from HTTP access logs.

All requests should be authenticated. The sender should specify an API key in the HTTP header named Api-Key. To acquire and manage your API keys, contact Moloco at contact@molocoads.com.

Resource  ¶

POST/api/ingestion/ad/{event}

Posts an ad-event such as impression or click. A request may contain detailed information only available at the context of placing a bid, such as the type of creative.

The following example shows how to post a ‘click’ event:

POST https://doublecheck.adsmoloco.com/api/ingestion/ad/click?android_idfa=ffffffff-ffff-4fff-8fff-ffffffffffff&account=moloco_account&product=moloco_app&happen_at=1514764800&network=moloco_dsp&ad_id=moloco-bidid1&ad_site=popular.app.com&sub_network=adx&ad_type=video&ad_format=interstitial

Resource  ¶

POST/api/ingestion/inapp/{event}

Posts an in-app event such as ‘install’. A request may contain information about the tracker’s attribution measurement. Unless the conversion is organic, information on the ad-event that the conversion is attributed to should be specified in the fields that have attr_to_ prefix.

If Moloco Doublecheck receives ad-events, only attr_to_network field is required, and, for more accurate analysis, attr_to_ad_id is optionally considered. With such information, Doublecheck is capable of joining datasets to retrieve all the necessary data.

If Moloco Doublecheck does not receive ad-events, however, comprehensive information on the ad-event is recommended (i.e. additional information should be provided in attr_to_ fields).

If the conversion is organic, information on the ad-event is ignored.

The following example shows how to post an ‘install’ event that is attributed to moloco_dsp's ‘click’ event:

POST https://doublecheck.adsmoloco.com/api/ingestion/inapp/install?android_idfa=ffffffff-ffff-4fff-8fff-ffffffffffff&account=moloco_account&product=moloco_app&happen_at=1514764800&attr_type=ct&attr_method=fingerprint&tracker=moloco_tracker&attr_to_happen_at=1514764800&attr_to_network=moloco_dsp&attr_to_ad_id=moloco-bidid1&attr_to_ad_site=popular.app.com&attr_to_sub_network=adx&attr_to_ad_type=video&attr_to_ad_format=interstitial
